59  // If true, this is a "poor quality" mode. The autoadjust
60  // code should always attempt to use a different mode to this
61  // mode in fullscreen.
62  //
63  // Some notes about what "poor quality" means in this context:
64  //
65  // The aspect ratio correction works by scaling up to the larger
66  // screen size and then drawing pixels on the edges between the
67  // "virtual" pixels so that an authentic blocky look-and-feel is
68  // achieved.
69  //
70  // For a mode like 640x480, you can imagine the grid of the
71  // "original" pixels spaced out, with extra "blurry" pixels added
72  // in the space between them to fill it out. However, when you're
73  // running at a resolution like 320x240, this is not the case. In
74  // the small screen case, every single pixel has to be a blurry
75  // interpolation of two pixels from the original image.
76  //
77  // If you run in 320x240 and put your face up close to the screen
78  // you can see this: it's particularly visible in the small yellow
79  // status bar numbers for example. Overall it still looks "okay"
80  // but there's an obvious - albeit small - deterioration in
81  // quality.
82  //
83  // Once you get to 640x480, all the original pixels are there at
84  // least once and it's okay (the higher the resolution, the more
85  // accurate it is). When I first wrote the code I was expecting
86  // that even higher resolutions would be needed before it would
87  // look acceptable, but it turned out to be okay even at 640x480.
88 
89  boolean poor_quality;
90 } screen_mode_t;
